Claudeのコードレビューで指摘された6つのポイントについて、現在のスクリプト（`backend_calc.py`）の状況を確認しました。ご指摘の通り、修正が必要な箇所が含まれています。

以下にそれぞれの問題点に対する修正の方針を提案します。

## Proposed Changes

### ① 列名マッチングが無音で失敗する可能性
**現状**: `df_kanken.columns.astype(str).str.strip()` では改行コード（`\n`）が除去されないため、エクセル側の「店舗\n在庫」といった列名を取りこぼす危険があります。
**提案**: 
改行コードを空文字に置換する処理を追加し、マッチングを確実に行うようにします。
```python
df_kanken.columns = df_kanken.columns.astype(str).str.replace('\n', '', regex=False).str.strip()
```

### ② 残り需要予測（Kankenファイル由来）が読んだ後に使われていない
**現状**: Kankenエクセルから「残り需要予測」を読み込んでいますが、最終的な計算ではスクリプト内で算出した「残り需要予測(現実ベース)」のみを使用しています。
**提案**:
意図的な切り替えであることを明確にするため、ソースコード上にコメントを追記します。
```python
# ※Kankenファイル由来の需要予測は参考値として読み込むが、
# 最終的な計算にはローリングキャストで算出した「残り需要予測(現実ベース)」を使用する
```

### ③ 期首在庫 = 倉庫在庫 + 店舗在庫 の意味論的な問題
**現状**: 「期首在庫」に店舗在庫を含めて「年末着地見込み」を計算しています。
**提案**:
計算結果自体は正しいものの、見た目の誤解を避けるためHTML出力側（またはバックエンドのコメント）で、「なぜ店舗在庫を含めるのか（店舗にある在庫も年末時点での全体残高の一部として扱うため）」を明記するコメントを追加します。

### ④ `actual_months_count` の判定ロジックに潜在的な脆弱性
**現状**: `df_stores["MTH"].nunique()` で月数をカウントしているため、データ内に「APR の合計」などの集計行が混入すると月数がズレる危険があります。
**提案**:
有効な月のリスト（JAN〜DEC）を定義し、それに合致するものだけをカウントするように防御的なコードに変更します。
```python
VALID_MONTHS = {'JAN', 'FEB', 'MAR', 'APR', 'MAY', 'JUN', 'JUL', 'AUG', 'SEP', 'OCT', 'NOV', 'DEC'}
actual_months_count = df_stores[df_stores["MTH"].isin(VALID_MONTHS)]["MTH"].nunique()
```

### ⑤ 119行のうち91行がゼロ（HTML出力への影響）
**現状**: マスタにあるすべてのカラーが出力されるため、実績や入荷予定がないカラーもHTML上に表示されてしまいます。
**提案**:
> [!NOTE]
> 以下のどちらの対応にするかご指定ください。
> - **A案（バックエンドで除外）**: スクリプトの最後で、「累計実績」や「FW入荷予定」などがすべて0の行をCSV/JSON出力前に除外する。
> - **B案（フロントエンドで対応）**: バックエンドは全データを出力し、HTML/JS側で「実績のあるものだけ表示」するトグルボタン等を設ける。
> （おすすめは**A案**です。データ量が減りスッキリします）

### ⑥ CUMULATIVE_PROGRESSが2024ベースという前提を明示すべき
**現状**: 2024年の進捗率を使っている旨のコメントはありますが、2026年も同じ季節パターンを適用しているという前提が少し弱いです。
**提案**:
後からコードを読む人のために、定数定義の付近に「2026年も2024年と同様の季節変動（売れ方のカーブ）を仮定して推計する」という旨のコメントを追記します。
